# Research Summary: Python CLI & SQLite Best Practices

## 1. CLI Library Comparison

### argparse (Standard Library)
- **Pros:** No dependencies, ships with Python, well-documented, sufficient for simple CLIs
- **Cons:** Verbose boilerplate, limited UX features, no automatic help formatting enhancements
- **Best for:** Simple scripts, minimal-dependency requirements

### Click
- **Pros:** Decorator-based, composable command groups, built-in testing utilities (`CliRunner`), wide ecosystem (Flask uses it), excellent error handling, supports lazy loading
- **Cons:** External dependency, slightly more complex for advanced patterns
- **Best for:** Medium-to-complex CLIs, projects needing rich UX without heavy dependencies
- **Key patterns:** `@click.group()`, `@click.command()`, `@click.option()`, `@click.argument()`

### Typer
- **Pros:** Built on Click, uses Python type hints for automatic CLI generation, minimal boilerplate, auto-generates help text, integrates with Pydantic for validation
- **Cons:** Newer library (less battle-tested), requires Python 3.6+, adds Typer+Click as dependencies
- **Best for:** Modern Python projects, rapid development, teams comfortable with type hints
- **Key patterns:** Function signatures define CLI interface; `typer.Typer()` app object

### Rich (Complementary Library)
- **Pros:** Beautiful terminal output (tables, progress bars, syntax highlighting, panels), integrates seamlessly with Click/Typer
- **Cons:** Adds dependency, overkill for simple scripts
- **Recommendation:** Pair with Typer or Click for production-quality UX

## 2. CLI UX Best Practices

### Command Structure
- Use **verb-noun** command naming: `task add`, `task list`, `task complete`, `task delete`
- Group related commands under a parent group (Click/Typer groups)
- Provide short aliases where sensible (`-v` for `--verbose`, `-n` for `--name`)

### User Feedback
- Always confirm destructive operations (deletions) with `--force` flag or interactive prompt
- Use exit codes correctly: `0` = success, `1` = general error, `2` = misuse
- Print human-readable success/error messages; use `stderr` for errors
- Support `--quiet` and `--verbose` flags for output control

### Help & Documentation
- Every command and option must have a `help=` string
- Include usage examples in `epilog` text
- Use `--version` flag via `@click.version_option()` or `typer`'s equivalent

### Input Validation
- Validate inputs early; fail fast with clear error messages
- Use Click's built-in types (`INT`, `Choice`, `Path`, `DateTime`) for automatic validation
- Never silently ignore bad input

### Configuration
- Support config file + environment variables + CLI flags (precedence: CLI > env > config file)
- Use `~/.config/appname/config.toml` or `~/.appname.toml` for user config
- Libraries: `python-dotenv` for env vars, `tomllib` (Python 3.11+) for TOML config

## 3. SQLite Integration Options

### sqlite3 (Standard Library)
- **Pros:** Zero dependencies, full SQL control, lightweight, ships with Python
- **Cons:** Manual schema management, no ORM, raw SQL strings, migration management is manual
- **Best for:** Simple apps, learning projects, when avoiding dependencies is critical
- **Key practices:**
  - Use `with conn:` context manager for automatic transaction commit/rollback
  - Always use parameterized queries (`?` placeholders) — **never string interpolation**
  - Enable WAL mode: `PRAGMA journal_mode=WAL` for better concurrent performance
  - Enable foreign keys: `PRAGMA foreign_keys = ON` (disabled by default in SQLite)
  - Use `row_factory = sqlite3.Row` for dict-like row access

### SQLAlchemy Core (No ORM)
- **Pros:** SQL expression language, schema definition in Python, migration support via Alembic, database-agnostic
- **Cons:** Heavier dependency, more setup
- **Best for:** Apps that might migrate to PostgreSQL/MySQL later

### SQLAlchemy ORM
- **Pros:** Full ORM with relationships, lazy loading, session management
- **Cons:** Significant overhead for simple CLIs, learning curve
- **Best for:** Complex domain models, larger applications

### Peewee
- **Pros:** Lightweight ORM, simpler than SQLAlchemy, good SQLite support, built-in migrations
- **Cons:** Less popular, smaller ecosystem
- **Best for:** Small-to-medium apps wanting an ORM without SQLAlchemy complexity

### **Recommendation for a Task Manager CLI:** Use `sqlite3` (standard library) with a thin repository/DAO layer, or Peewee if ORM features are desired.

## 4. SQLite Schema Design for a Task Manager

### Recommended Schema

```sql
-- Core tasks table
CREATE TABLE IF NOT EXISTS tasks (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    title       TEXT NOT NULL CHECK(length(title) > 0),
    description TEXT,
    status      TEXT NOT NULL DEFAULT 'pending'
                CHECK(status IN ('pending', 'in_progress', 'done', 'cancelled')),
    priority    INTEGER NOT NULL DEFAULT 2
                CHECK(priority BETWEEN 1 AND 5),  -- 1=lowest, 5=highest
    due_date    TEXT,  -- ISO 8601: 'YYYY-MM-DD'
    created_at  TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at  TEXT NOT NULL DEFAULT (datetime('now')),
    project_id  INTEGER REFERENCES projects(id) ON DELETE SET NULL,
    tags        TEXT  -- JSON array string: '["work","urgent"]'
);

-- Optional projects table
CREATE TABLE IF NOT EXISTS projects (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    name        TEXT NOT NULL UNIQUE,
    description TEXT,
    created_at  TEXT NOT NULL DEFAULT (datetime('now'))
);

-- Recommended indexes
CREATE INDEX IF NOT EXISTS idx_tasks_status   ON tasks(status);
CREATE INDEX IF NOT EXISTS idx_tasks_due_date ON tasks(due_date);
CREATE INDEX IF NOT EXISTS idx_tasks_project  ON tasks(project_id);

-- Trigger to auto-update updated_at
CREATE TRIGGER IF NOT EXISTS update_tasks_timestamp
AFTER UPDATE ON tasks
BEGIN
    UPDATE tasks SET updated_at = datetime('now') WHERE id = NEW.id;
END;
```

### Schema Design Principles
- **Use TEXT for dates** in ISO 8601 format — SQLite has no native DATE type; ISO strings sort correctly
- **Use CHECK constraints** for enums (status, priority) rather than separate lookup tables for simplicity
- **Store tags as JSON** for flexibility without a junction table (acceptable for small-scale apps)
- **Always include `created_at`/`updated_at`** audit columns
- **Enable foreign keys** explicitly at connection time
- **Use AUTOINCREMENT** sparingly — plain `INTEGER PRIMARY KEY` is sufficient unless you need guaranteed non-reuse of IDs

### Migration Strategy
- For simple apps: version the schema with a `schema_version` table or `PRAGMA user_version`
- For growing apps: use **Alembic** (with SQLAlchemy) or **yoyo-migrations** (raw SQL)
- Always run schema creation with `CREATE TABLE IF NOT EXISTS` for idempotency

## 5. Project Structure Best Practices

```
task_manager/
├── pyproject.toml          # Build config, dependencies (use Poetry or Hatch)
├── README.md
├── src/
│   └── task_manager/
│       ├── __init__.py
│       ├── cli.py          # Click/Typer command definitions
│       ├── db.py           # Database connection & initialization
│       ├── models.py       # Data classes / schema definitions
│       ├── repository.py   # Data access layer (CRUD operations)
│       └── config.py       # App configuration management
└── tests/
    ├── test_cli.py         # Use Click's CliRunner or Typer's test client
    ├── test_repository.py  # Test DB layer with in-memory SQLite (:memory:)
    └── conftest.py         # Pytest fixtures (temp DB, etc.)
```

### Key Architectural Patterns
- **Repository Pattern:** Separate data access from CLI logic; `repository.py` contains all SQL
- **Dependency Injection:** Pass DB connection to repository; makes testing easy
- **In-memory SQLite for tests:** `sqlite3.connect(':memory:')` — fast, isolated tests
- **Single DB connection per command:** Open connection at command start, close at end (or use context manager)

## 6. Packaging & Distribution

- Use **`pyproject.toml`** (PEP 517/518) with Poetry, Hatch, or Flit
- Define CLI entry point:
  ```toml
  [project.scripts]
  task = "task_manager.cli:app"  # Typer

  task = "task_manager.cli:main"  # Click
  ```

- Use **`pipx`** for installation to isolate CLI tools in their own virtualenvs
- Consider **PyInstaller** or **Nuitka** for single-binary distribution

## 8. Areas of Uncertainty / Follow-up Research

- **Typer maturity:** Typer is actively developed but has had breaking changes between versions; pin versions carefully
- **Async SQLite:** If async support is needed, investigate `aiosqlite` — not relevant for simple CLIs but worth noting
- **Tag storage:** JSON string vs. junction table trade-off depends on query needs; if filtering by tag is frequent, a proper `task_tags` junction table is preferable
- **Windows compatibility:** Rich and Click both handle Windows terminals, but test ANSI color support on older Windows versions
