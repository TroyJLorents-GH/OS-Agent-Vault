# Research Findings: Episodic Memory in AI Agents

## Overview

Episodic memory in AI agents refers to the ability to store, retrieve, and reason over specific past interactions, events, and experiences — analogous to human episodic memory (the "what, where, when" of personal experience). This is distinct from **semantic memory** (general knowledge), **procedural memory** (how to do things), and **working memory** (immediate context).

## Section 1: Conceptual Foundation

### What Episodic Memory Means for AI Agents

LLMs are stateless by default — each inference call has no inherent memory of prior interactions. Episodic memory systems solve this by:

- **Persisting interaction history** beyond the context window
- **Selectively retrieving** relevant past episodes at inference time
- **Summarizing or compressing** older memories to manage storage
- **Indexing events temporally and semantically** for efficient lookup

The core challenge is the **context window bottleneck**: LLMs can only attend to a finite amount of text, so episodic memory systems must decide *what to remember*, *how to store it*, and *when to retrieve it*.

## Section 2: MemGPT

### Background
MemGPT (Memory-GPT) was introduced in a 2023 paper by Packer et al. from UC Berkeley. It drew inspiration from **operating system virtual memory** — the idea that a hierarchy of storage tiers can give the illusion of unlimited memory.

### Architecture
MemGPT uses a **tiered memory hierarchy**:

| Tier | Name | Description |
|------|------|-------------|
| L1 | **In-Context (Working) Memory** | The active LLM context window — fast, limited |
| L2 | **External Storage (Recall/Archival)** | Persistent database outside the context window |

**Recall Storage** holds recent conversation history (episodic).
**Archival Storage** holds long-term facts and documents (semantic/episodic).

### How Episodic Memory Works in MemGPT
- The LLM agent is given **function-calling tools** to explicitly manage memory:
  - `memory_search()` — query past episodes via vector similarity
  - `memory_append()` / `memory_replace()` — write to in-context memory blocks
  - `archival_memory_insert()` / `archival_memory_search()` — interact with long-term storage
- The agent **self-directs** memory operations — it decides when to save or retrieve
- Memory is stored as **vector embeddings** in a database (e.g., ChromaDB, PostgreSQL + pgvector)
- **Recursive summarization** compresses older conversation turns to free context space

### Key Innovation
The agent itself is the memory manager — it reasons about what to remember, acting as its own OS scheduler for memory resources.

### Limitations
- Relies on the LLM making good decisions about when to invoke memory tools
- Latency overhead from self-directed tool calls
- Memory quality depends heavily on model capability

## Section 3: Zep

### Background
Zep is a **commercial memory layer** (with an open-source community edition) designed specifically as infrastructure for AI agents and chatbots. It is model-agnostic and operates as a separate service.

### Architecture
Zep uses a **knowledge graph + vector store hybrid** approach:

- **Episodic Memory**: Raw message history stored as a session-based timeline
- **Semantic Memory**: Facts extracted from conversations stored in a **knowledge graph** (using Neo4j or similar)
- **Entity Memory**: Named entities (people, places, organizations) tracked across sessions

### How Episodic Memory Works in Zep
1. **Session-based storage**: Conversations are stored per user/session with timestamps
2. **Automatic summarization**: Zep runs background processes to summarize older message windows (using an LLM), reducing storage while preserving episodic gist
3. **Hybrid retrieval**: Combines:
   - **BM25 (keyword)** search over message history
   - **Vector similarity** search over embeddings
   - **Graph traversal** for entity-linked memories
4. **Temporal awareness**: Memories are explicitly timestamped; the system can answer "what did the user say last Tuesday?"
5. **Zep Cloud (v2)** introduced a **graph-based memory** called the *Graphiti* engine, which builds a dynamic temporal knowledge graph from conversations

### Key Innovation
Zep separates memory infrastructure from the agent — it's a **memory-as-a-service** model. The agent doesn't manage memory; Zep does it automatically in the background.

### Graphiti (Zep's Graph Engine)
- Builds a **bi-temporal knowledge graph**: tracks both *when something happened* and *when it was recorded*
- Supports **contradiction resolution** (if a user says something different later, the graph updates)
- Enables complex queries like "what were the user's preferences before they changed their mind?"

### Limitations
- Adds infrastructure dependency (separate service)
- Knowledge graph extraction quality depends on LLM accuracy
- Commercial product with usage costs at scale

## Section 4: Letta (formerly MemGPT)

### Background
**Letta** is the evolution and productization of the MemGPT research project. The MemGPT team rebranded/expanded into Letta (the company and framework) in late 2023/2024. Letta is both:
- An **open-source agent framework** (letta-ai/letta on GitHub)
- A **hosted platform** for stateful agents

### How Letta Extends MemGPT's Episodic Memory

Letta retains MemGPT's core memory architecture but adds significant productization:

| Feature | MemGPT (Research) | Letta (Production) |
|---------|------------------|-------------------|
| Memory tiers | In-context + archival | Same, plus structured memory blocks |
| Storage backends | ChromaDB, basic DBs | PostgreSQL, pgvector, scalable backends |
| Multi-agent | Limited | First-class multi-agent memory sharing |
| Memory blocks | Basic | **Typed, named memory blocks** with schemas |
| API | Research code | REST API + Python/TypeScript SDKs |
| Persistence | File-based | Full database persistence |

### Letta-Specific Memory Innovations

1. **Named Memory Blocks**: Memory is organized into typed blocks (e.g., `human`, `persona`, `system`) that persist across sessions and can be updated atomically
2. **Shared Memory**: Multiple agents can read/write shared memory blocks — enabling **collaborative agent memory**
3. **Memory Tools as First-Class Citizens**: Tool calls for memory are built into the agent schema, not bolted on
4. **Agent State Persistence**: Entire agent state (memory + conversation + tool definitions) is serialized and stored, enabling true statefulness

### Relationship to MemGPT
Letta IS MemGPT, matured. The research prototype became a production framework. The core episodic memory model (OS-inspired tiered memory with self-directed management) remains the philosophical foundation.

## Section 5: Comparative Analysis

### Architectural Philosophy

| Dimension | MemGPT | Zep | Letta |
|-----------|--------|-----|-------|
| **Memory model** | OS virtual memory metaphor | Knowledge graph + vector hybrid | Extended OS metaphor (production-grade) |
| **Who manages memory** | The agent itself | External service (automatic) | The agent itself (with better tooling) |
| **Episodic storage** | Vector DB (recall store) | Session timeline + graph | Vector DB + structured blocks |
| **Retrieval mechanism** | Agent-initiated tool calls | Automatic + on-demand API | Agent-initiated tool calls |
| **Temporal reasoning** | Limited (recency ordering) | Strong (bi-temporal graph) | Moderate |
| **Cross-session memory** | Yes (archival) | Yes (persistent sessions) | Yes (persistent agent state) |
| **Multi-agent** | No | Partial | Yes (shared memory blocks) |
| **Deployment model** | Self-hosted library | Service (OSS + Cloud) | Self-hosted + Cloud |
| **Memory agency** | High (agent decides) | Low (automatic) | High (agent decides) |

### Retrieval Strategy Comparison

**MemGPT/Letta**: Pull-based. The agent explicitly calls memory search tools when it determines retrieval is needed. This is powerful but depends on the agent recognizing when it *should* remember.

**Zep**: Push-based + Pull-based hybrid. Zep automatically injects relevant memories into context (push) AND exposes APIs for explicit retrieval (pull). This reduces reliance on agent judgment.

### Memory Compression Strategies

- **MemGPT/Letta**: Recursive summarization triggered by context overflow — older messages are summarized and moved to archival storage
- **Zep**: Background summarization pipeline runs asynchronously; the agent is never blocked by compression

## Section 7: Open Research Challenges

1. **Memory Relevance**: How to reliably retrieve the *right* episodic memory, not just the most similar one
2. **Memory Decay and Forgetting**: Should agents forget? How to implement principled forgetting
3. **Contradiction Handling**: When new episodes contradict old ones, which takes precedence?
4. **Hallucinated Memories**: LLMs can confabulate memories that weren't stored
5. **Scalability**: Millions of users × long conversation histories = massive storage and retrieval challenges
6. **Privacy**: Episodic memory contains sensitive personal data; deletion and access control are hard
7. **Evaluation**: No standard benchmarks for episodic memory quality in agents (MemGPT paper used custom evaluations)

## Key Takeaways

1. **MemGPT pioneered** the OS-inspired, self-directed episodic memory model for LLM agents
2. **Letta is MemGPT productized** — same philosophy, production engineering, multi-agent capabilities
3. **Zep takes a different philosophy** — memory as infrastructure, automatic management, graph-based temporal reasoning
4. **The core trade-off** is *agent-directed* (MemGPT/Letta) vs. *infrastructure-directed* (Zep) memory management
5. **Hybrid retrieval** (vector + keyword + graph) is emerging as the standard for robust episodic recall
6. **Temporal knowledge graphs** (Zep's Graphiti) represent a frontier approach for episodic memory with strong temporal reasoning
